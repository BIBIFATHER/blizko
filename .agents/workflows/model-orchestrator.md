# Model Orchestrator

## Жесткое правило

Перед каждой крупной задачей сначала классифицируй работу:

1. `deep_code` — сложная реализация, архитектура, multi-file refactor, hard debugging
2. `fast_code` — простые правки, rename, wiring, boilerplate, cleanup
3. `research` — свежие данные, рынок, конкуренты, цены, SEO, документы
4. `codebase_review` — полный анализ кода, аудит кодовой базы, quality/security/release review
5. `writing` — UX copy, маркетинг, narrative, positioning
6. `strategy` — продуктовая аналитика, GTM, decision memo, roadmap

Если текущая модель не оптимальна для класса задачи, **сначала предложи переключение**, а уже потом выполняй работу.

Не предлагай переключение ради переключения. Предлагай только если это даёт:

- выше качество результата
- заметную экономию токенов
- доступ к свежим данным
- лучшее удержание длинного контекста

## Когда НЕ нужно переключение

- задача уже почти завершена и контекст слишком дорогой для переноса
- нужна точечная правка в 1-2 файла без аналитики
- разница между моделями не даст явного выигрыша

## Когда предлагать переключение

1. **Задача аналитическая** → «💡 Gemini 3.1 Pro (High) лучше справится с анализом»
2. **Нужны свежие данные** → «💡 Perplexity найдёт актуальные цифры»
3. **Креатив/тексты** → «💡 GPT 5.3 сильнее в копирайтинге»
4. **Простая механика** → «💡 Gemini Flash справится за 10% стоимости»
5. **Сложный код** → оставайся на Claude Opus 4.6
6. **Полный анализ кодовой базы** → не маршрутизируй как generic analytics; это review-задача

## Формат предложения

Используй короткий блок перед началом работы:

```
💡 Для [задача] лучше переключиться на [модель]:
— [причина 1]
— [причина 2]
Экономия: ~[X]x по токенам
Хотите переключиться?
```

Если переключение не требуется, молча продолжай.

Не превращай предложение о переключении в длинное эссе. Максимум 2 причины и 1 строка про выигрыш.

## Передача контекста

При переключении подготовь промпт:

- Что уже сделано (кратко)
- Текущая задача
- Ключевые файлы
- Ожидаемый формат результата

## Готовые рекомендации по умолчанию

- `deep_code` → Claude Opus 4.6
- `fast_code` → Gemini 3 Flash
- `codebase_review` → Claude Opus 4.6
- `research` → Perplexity или Gemini 3.1 Pro (High)
- `writing` → GPT 5.3
- `strategy` → Gemini 3.1 Pro (High)

## Execution rule

После предложения о model switch не зависай в длинном ожидании, если текущая модель всё ещё приемлема.

- если текущая модель допустима, можно начать работу сразу
- если без другой модели задача сильно потеряет в качестве, явно остановись и запроси переключение

## Research-only rule

Если запрос пользователя чисто research / market / competitor / pricing:

- не смешивай его с кодом, фичами и bugfix-задачами
- ответ должен быть только:
  - `Objective`
  - `Model switch`
  - при необходимости одна строка `Жду переключения`

Не добавляй delegation plan для unrelated задач в том же ответе.

## Research output contract

Для research-задач итог должен быть структурирован так:

1. `Facts`
2. `Inferences`
3. `Strategic implication`
4. `Open questions`

### Rules

- факты отделяй от выводов явно
- если данных/источников нет, не маскируй inference под факт
- для цен, рынка, конкурентов и трендов по возможности указывай источник
- если работа выполнена без live search, прямо отмечай, что часть выводов требует веб-подтверждения
- для рынка, конкурентов, цен, SEO и текущих продуктов каждый блок `Facts` должен содержать источник
- если источник не подтверждён в этой сессии, пиши: `Источник не подтверждён в этой сессии`
- в `Inferences` не опирайся на неподтверждённые цифры как на установленный факт

### Anti-pattern

Не делай research в формате:
- длинный маркетинговый нарратив
- смесь фактов и мнений без маркировки
- сильные рыночные выводы без указания степени уверенности
- факты без ссылок там, где ссылки ожидаемы

## Full Codebase Review Rule

Если пользователь пишет:

- `проведи полный анализ кода`
- `сделай полный аудит кодовой базы`
- `полный code review`
- `оцени качество всей кодовой базы`

то классифицируй задачу как `codebase_review`, а не как generic `analysis`.

### Что запрещено считать достаточным

- только `eslint`
- только `tsc`
- только `build`
- только обзор конфигов
- сведение результата к одному broken import или количеству warning'ов

### Минимальный объём проверки

1. ключевые entrypoints и app shell
2. core services / data flow
3. API / auth / security sensitive paths
4. state / effects / UI critical screens
5. tests and verification posture
6. env / runtime / release dependencies

### Обязательный формат ответа

1. `Findings`
2. `Open questions / assumptions`
3. `Architecture notes`
4. `Verification coverage`
5. `Recommended fix order`

### Правила

- findings first, ordered by severity
- file references are mandatory for concrete issues
- clearly separate verified facts from inference
- explicitly state what was not checked
- if the pass is only static analysis, say `Это не полный анализ кодовой базы; это только static analysis`

### Anti-pattern

Не отвечай на полный анализ кодовой базы так:

- `линтер чистый, есть N warnings`
- `нашёл один TS import`
- `сохранил analysis_results.md`
- `хочешь, я теперь реально посмотрю код`
