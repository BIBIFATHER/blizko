# Design Rules

Цель: удерживать Blizko в одном trust-first visual language и не скатываться обратно в набор случайных карточек.

## Базовый принцип
- Интерфейс должен выглядеть как спокойный premium care-product, а не как generic startup UI.
- Основа: тёплые поверхности, мягкая глубина, ясные CTA, редакционная типографика, доказательное доверие.
- Каждый экран должен отвечать на три вопроса:
  - что происходит сейчас
  - почему этому можно доверять
  - какое следующее действие полезнее всего

## Обязательный каркас для ключевых экранов
- `hero-shell`:
  - использовать для первого экрана или главного смыслового блока
  - внутри держать не больше двух колонок: narrative + stats/context
- `section-shell`:
  - использовать для grouped-content секций
  - если блок не требует отдельной иерархии, не городить новый визуальный стиль
- `form-footer-rail`:
  - использовать для финального CTA или reassurance-блока
- `Card`:
  - использовать для внутренних единиц внутри section/hero
  - не заменять им layout целого экрана

## Типографика
- `h1/h2/h3` и display-copy должны жить в serif/display-ритме, уже заданном в `index.css`
- body copy должен быть спокойным, читаемым, с нормальным line-height
- нельзя возвращаться к “фиолетовый жирный hero + случайный small text”

## Цвет и поверхности
- не использовать фиолетовый как primary brand-accent
- главный акцент: honey / warm stone / muted emerald / soft sky
- цвет должен усиливать доверие и контекст, а не кричать
- каждая новая поверхность должна опираться на существующие классы:
  - `hero-shell`
  - `section-shell`
  - `hero-stat`
  - `surface-panel`
  - `trust-badge`

## CTA
- на экран максимум один primary CTA
- secondary CTA допустим только если он реально помогает принять решение без давления
- CTA должен быть выровнен по смыслу:
  - primary = движение вперед
  - secondary = уточнить, вернуться, обсудить

## Trust Layer
- trust-сигналы должны быть встроены в контент, а не висеть декоративно
- использовать trust-badges, proof-cards, verified-stats и human-readable explanations
- если экран влияет на решение пользователя, там обязан быть explainability-layer

## Что запрещено
- случайные градиенты без системной роли
- новая локальная цветовая схема на отдельном экране
- перегрузка мелкими плашками и pill-чипами
- heavy glassmorphism ради “вау”
- layout, где каждое действие выглядит одинаково важным

## Приоритет экранов
- сначала customer-facing journey
- потом dashboards и account surfaces
- потом вторичные модалки и служебные экраны
- админка последней, если она не влияет на доверие пользователя
