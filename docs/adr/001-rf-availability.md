# ADR-001: Доступность blizko.app из РФ без VPN

**Статус:** Accepted  
**Дата:** 2026-06-03  
**Автор:** Антон Аносов  
**Linear:** BLI-85

---

## Контекст

blizko.app размещён на Vercel. DNS проксируется через Cloudflare (настроено в рамках BLI-35, май 2026). Диагностика показала: HTML-документ доходит до пользователя (встроенный splash рендерится), но JS-бандл не загружается на части российских мобильных сетей (Tele2, некоторые MVNO). С VPN проблема исчезает — следовательно, код в порядке, проблема сетевая.

**Что подтверждено:**
- При наличии CF proxy HTML доходит (возможно, из CF cache).
- JS-ассеты (`/assets/*.js`) после каждого деплоя дают cache miss на CF → запрос идёт на Vercel origin → трансатлантический roundtrip не проходит на проблемных операторах.
- Google Fonts (`fonts.googleapis.com`) дополнительно блокирует render, если домен недоступен.

---

## Варианты решения

### Вариант A — Оптимизация кеширования на CF (выбран сейчас)
Добавить явные `Cache-Control: public, max-age=31536000, immutable` заголовки для `/assets/*` в `vercel.json`. Vite использует content-hash в именах файлов, поэтому long cache безопасен. После первого cache miss CF кеширует бандлы на российских edge-узлах и отдаёт последующим пользователям уже из России.

**Плюсы:** Не требует инфраструктуры, деплоится вместе с кодом.  
**Минусы:** Первый пользователь после деплоя всё ещё получает медленный ответ; не помогает операторам, которые блокируют CF.

### Вариант B — Nginx reverse proxy на российском VPS (резервный)
VPS (Timeweb / Selectel, ~500 руб/мес) принимает трафик, проксирует на Vercel. В Cloudflare origin меняется с Vercel на IP российского сервера. CF остаётся для DDoS-защиты и TLS-терминации.

**Плюсы:** Обходит проблему для всех российских операторов; nginx кеширует ассеты локально.  
**Минусы:** Дополнительный сервер, его нужно обслуживать; latency для нероссийских пользователей (незначительно — целевая аудитория РФ).

Config готов: `infra/nginx-rf-proxy.conf`

### Вариант C — Полный переезд на RU-инфраструктуру
Статика → Yandex Cloud CDN / VK Cloud. API → Timeweb serverless или VPS. Supabase → self-hosted.

**Плюсы:** Максимальная надёжность.  
**Минусы:** Трудоёмко (недели работы), не нужно на стадии пилота.

---

## Решение

**Сейчас:** Вариант A + исправление неблокирующей загрузки Google Fonts + fix SW pre-cache.

**Триггер для перехода к Варианту B:**  
Если после деплоя Варианта A smoke-тест из РФ (DevTools Network → `CF-Cache-Status`) показывает, что JS-бандлы всё ещё не загружаются для части операторов — разворачиваем Nginx-прокси на Timeweb.

**Вариант C** — только если объём трафика или регуляторные требования (152-ФЗ, локализация данных) сделают это необходимым.

---

## Изменения в коде (Вариант A)

| Файл | Изменение |
|------|-----------|
| `vercel.json` | Cache-Control headers для `/assets/*`, иконок и `/` |
| `index.html` | Async Google Fonts (preload → onload) |
| `public/sw.js` | Убран `/` из PRE_CACHE |

---

## Проверка после деплоя

1. `npm run build` — зелёный.
2. После деплоя в production: открыть Chrome DevTools → Network → загрузить страницу дважды → проверить, что JS-бандлы получают заголовок `CF-Cache-Status: HIT`.
3. Smoke-тест из РФ (телефон без VPN): `blizko.app` → форма заявки открывается без белого экрана.
4. Если smoke показывает проблему → развернуть Вариант B по инструкции в `infra/nginx-rf-proxy.conf`.
